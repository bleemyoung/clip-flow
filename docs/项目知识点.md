# ClipFlow 项目知识点

## 1. Electron 三层职责划分

### 主进程 `main`

- 负责窗口创建
- 负责桌面能力接入
- 负责后续剪贴板监听、文件读写和本地持久化

### 预加载层 `preload`

- 负责桥接渲染层可调用的安全 API
- 通过 `contextBridge` 向 `window` 暴露受控能力
- 避免在渲染层直接打开 Node 能力

### 渲染层 `renderer`

- 负责页面 UI、交互状态和用户操作
- 当前采用 `Vue3` 组件化组织结构

## 2. `electron-vite` 入口配置要点

当前项目需要分别配置三类入口：

- `main.build.lib.entry`
- `preload.build.lib.entry`
- `renderer.build.rollupOptions.input`

如果缺少任一入口，开发或构建阶段都可能报错，尤其是 `electron-vite@3.x` 对入口声明比较明确。

## 3. preload 为什么容易引发白屏

preload 脚本属于 Electron 启动时非常靠前的一层，如果它加载失败，渲染层中依赖 preload 暴露的 API 会直接报错。

当前项目中典型链路是：

1. preload 通过 `contextBridge.exposeInMainWorld('clipflow', api)` 暴露 API
2. 渲染层 `App.vue` 调用 `window.clipflow.getAppMeta()`
3. 如果 preload 失败，`window.clipflow` 不存在，页面就可能白屏

## 4. 为什么 preload 需要用 CJS

本项目当前构建方式下，preload 作为 Electron 直接执行的脚本，使用 CJS 更稳妥。此前默认输出为 ESM 时，出现了：

```text
Cannot use import statement outside a module
```

因此当前方案中将 preload 输出固定为：

- `format: 'cjs'`
- `entryFileNames: 'index.js'`

## 5. 当前项目分层设计

### `main`

放 Electron 主进程逻辑，当前已完成窗口创建和开发态调试支持。

### `preload`

放桥接层逻辑，当前已暴露最小 `getAppMeta()` API。

### `renderer`

放 Vue 页面层，当前已完成：

- 主界面布局
- 三模块切换
- mock 数据展示
- 文本转换纯前端逻辑

### `shared`

放跨层共享内容，当前包括：

- 常量
- 领域类型
- IPC 类型

## 7. 阶段 3 计划知识点

当前阶段 3 尚未实施代码，但已确定后续采用的技术方向如下：

### Electron `clipboard.readText()`

- 适合当前 MVP 阶段做文本剪贴板轮询
- 实现简单，利于先验证剪贴板历史主链路

### 单 JSON 文件持久化

- 适合当前阶段的轻量本地存储
- 能满足历史记录保留与重启恢复
- 先保证稳定和简单，后续再评估是否需要升级存储方案

### `preload` 受控 API 与订阅模式

- `preload` 层适合向渲染层暴露受控读取和操作能力
- 当主进程历史记录变化时，可由桥接层提供订阅式更新入口
- 这样可以避免渲染层直接依赖系统能力

### `renderer` 只负责展示与操作

- 渲染层只负责列表展示、按钮交互和状态呈现
- 不直接访问系统剪贴板和本地文件
- 系统能力和持久化应继续留在 `main` 与 `preload` 层

## 8. 开发态新增 `preload` / IPC 能力时要关注会话刷新

当 Electron 项目在开发态新增 `preload` 暴露方法或 `ipcMain.handle(...)` 处理器时，不能默认认为仅刷新渲染页面就能拿到新能力。

典型影响：

- `renderer` 代码已经更新，但 `main` 或 `preload` 仍可能停留在旧会话
- `window.clipflow` 上的新方法未生效，或对应 IPC channel 尚未注册
- 页面上表现为“按钮逻辑看起来没问题，但运行时调用失败”

这类问题的排查顺序建议是：

1. 先确认 `preload` 是否真的暴露了新方法
2. 再确认 `main` 是否注册了对应 IPC handler
3. 最后确认当前开发进程是否已完整重启，而不是只刷新渲染页面

对应到 `ClipFlow` 的经验是：

- 系统能力优先继续走 `main -> preload -> renderer` 的受控链路
- 但在开发态联调阶段，可以为纯复制这类轻能力增加最小兜底，避免因为旧会话导致误判业务逻辑失败
- 页面提示最好只表达用户可理解结果，同时在排障文档中单独记录真实根因

## 9. 两种文本复制链路的底层差异

当前 `ClipFlow` 的文本复制可以理解为两种实现路径：

### `preload / Electron` 链路

调用路径：

`renderer -> preload -> ipcRenderer -> main -> electron.clipboard.writeText()`

在项目中的典型流程是：

1. 渲染层调用 `window.clipflow.copyText(text)`
2. `preload` 通过 `ipcRenderer.invoke(...)` 转发给主进程
3. 主进程通过 `ipcMain.handle(...)` 接收请求
4. 最终由 `electron.clipboard.writeText(text)` 写入系统剪贴板

特点：

- 属于 Electron 应用内部的受控桌面能力调用
- 更符合 `main / preload / renderer` 分层
- 适合作为正式主链路
- 依赖 `preload` 注入、IPC 注册和当前 Electron 会话完整刷新

### `navigator.clipboard` 链路

调用路径：

`renderer -> Chromium Clipboard API -> OS Clipboard`

典型流程是：

1. 渲染层直接调用 `navigator.clipboard.writeText(text)`
2. Chromium 的 Web Clipboard API 直接处理复制
3. 最终写入系统剪贴板

特点：

- 属于页面直接调用浏览器能力
- 不经过 `preload` 与 `ipcMain / ipcRenderer`
- 链路更短，更适合作为轻量兜底
- 和当前项目的 Electron 分层风格不完全一致

当前项目的推荐结论是：

- 正式主链路优先使用 `preload / Electron`
- `navigator.clipboard` 仅作为开发态或轻场景兜底
