# ClipFlow 项目知识点

## 1. Electron 三层职责划分

### 主进程 `main`

- 负责窗口创建
- 负责桌面能力接入
- 负责后续剪贴板监听、文件读写和本地持久化

### 预加载层 `preload`

- 负责桥接渲染层可调用的安全 API
- 通过 `contextBridge` 向 `window` 暴露受控能力
- 避免在渲染层直接打开 Node 能力

### 渲染层 `renderer`

- 负责页面 UI、交互状态和用户操作
- 当前采用 `Vue3` 组件化组织结构

## 2. `electron-vite` 入口配置要点

当前项目需要分别配置三类入口：

- `main.build.lib.entry`
- `preload.build.lib.entry`
- `renderer.build.rollupOptions.input`

如果缺少任一入口，开发或构建阶段都可能报错，尤其是 `electron-vite@3.x` 对入口声明比较明确。

## 3. preload 为什么容易引发白屏

preload 脚本属于 Electron 启动时非常靠前的一层，如果它加载失败，渲染层中依赖 preload 暴露的 API 会直接报错。

当前项目中典型链路是：

1. preload 通过 `contextBridge.exposeInMainWorld('clipflow', api)` 暴露 API
2. 渲染层 `App.vue` 调用 `window.clipflow.getAppMeta()`
3. 如果 preload 失败，`window.clipflow` 不存在，页面就可能白屏

## 4. 为什么 preload 需要用 CJS

本项目当前构建方式下，preload 作为 Electron 直接执行的脚本，使用 CJS 更稳妥。此前默认输出为 ESM 时，出现了：

```text
Cannot use import statement outside a module
```

因此当前方案中将 preload 输出固定为：

- `format: 'cjs'`
- `entryFileNames: 'index.js'`

## 5. 当前项目分层设计

### `main`

放 Electron 主进程逻辑，当前已完成窗口创建和开发态调试支持。

### `preload`

放桥接层逻辑，当前已暴露最小 `getAppMeta()` API。

### `renderer`

放 Vue 页面层，当前已完成：

- 主界面布局
- 三模块切换
- mock 数据展示
- 文本转换纯前端逻辑

### `shared`

放跨层共享内容，当前包括：

- 常量
- 领域类型
- IPC 类型
